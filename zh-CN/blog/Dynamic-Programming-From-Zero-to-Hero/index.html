рЯОѓ What is Dynamic Programming?вАЛ

Dynamic Programming (DP) is a method for solving complex problems by breaking them down into simpler subproblems, solving each subproblem just once, and storing their solutions.

Three key concepts:

  1. A problem can be broken into smaller subproblems
  2. Those subproblems repeat
  3. We can store their answers and reuse them

рЯФС OneвАСsentence definition:

Dynamic Programming = Recursion + Memory

рЯТ° Simple Analogy: Imagine climbing stairs. To reach step 10, you need to know how to reach step 9. To reach step 9, you need step 8... DP remembers solutions to these smaller steps!

рЯУК The Core IdeaвАЛ

Solving big problem
    вЖУ
Break into smaller problems
    вЖУ
Solve & remember solutions
    вЖУ
Reuse solutions to solve bigger problem

Key Properties of DP Problems:вАЛ

  1. Optimal Substructure: Optimal solution can be constructed from optimal solutions of subproblems

    • The "Optimal" solution to the big problem can be constructed from the "Optimal" solutions of its subproblems.
    • If you want the shortest path from A to C, and it goes through B, the path must include the shortest path from A to B.

  2. Overlapping Subproblems: Same subproblems are solved multiple times

    • This means the problem can be broken down into smaller versions of itself, and those smaller versions repeat.

рЯФД Two Approaches to DP (Top-Down vs Bottom-Up)вАЛ

There are two ways to implement DP. Think of them as "The Lazy Way" and "The Organized Way."

Method 1: Top-Down (Memoization)вАЛ

You start with the big problem and break it down. If you've solved a subproblem before, you look up the answer in your "notebook" (usually a dictionary or array).

  • Vibe: "I'll solve it when I need it, and save the answer for later."
  • Tool: Recursion + Cache.
flowchart LR
    Call --> CheckCache
    CheckCache -->|Found| Return
    CheckCache -->|Not Found| Compute
    Compute --> Save
    Save --> Return
memo = {}

def fib(n):
    if n <= 1:
        return n
    if n in memo:
        return memo[n]
    memo[n] = fib(n-1) + fib(n-2)
    return memo[n]

Method 2: Bottom-Up (Tabulation)вАЛ

You start with the smallest possible subproblems (base cases) and fill out a table until you reach the goal.

  • Vibe: "Let's build a foundation from the ground up."
  • Tool: Iteration (Loops) + Table (Array/Grid).
flowchart LR
    Base --> DP1 --> DP2 --> DP3 --> Answer
def fib(n):
    dp = [0] * (n + 1)
    dp[1] = 1
    for i in range(2, n + 1):
        dp[i] = dp[i-1] + dp[i-2]
    return dp[n]

вЬЕ Pros: No recursion, often faster

рЯФС Two Key Properties of DP (Must Memorize)вАЛ

1. Overlapping SubproblemsвАЛ

Same subproblem appears again and again

graph TD
    A["fib(5)"] --> B["fib(4)"]
    A --> C["fib(3)"]
    B --> C

2. Optimal SubstructureвАЛ

The optimal solution of a problem depends on optimal solutions of subproblems

Examples:

  • Shortest path
  • Maximum profit
  • Minimum steps

рЯІ© Example 1: Fibonacci NumbersвАЛ

Problem: Find the nth Fibonacci numberвАЛ

fib(0) = 0
fib(1) = 1
fib(n) = fib(n-1) + fib(n-2) for n > 1

Call Tree ExplosionвАЛ

flowchart TD
    F5 --> F4
    F5 --> F3
    F4 --> F3
    F4 --> F2
    F3 --> F2
    F3 --> F1

вЭМ Problem: fib(3) and fib(2) are computed multiple times

1. Brute Force (Recursive) - вЭМ InefficientвАЛ

def fib_naive(n):
    if n <= 1:
        return n
    return fib_naive(n-1) + fib_naive(n-2)
# Time: O(2вБњ) - Exponential!

2. Memoization (Top-Down DP) - вЬЕ EfficientвАЛ

def fib_memo(n, memo={}):
    if n <= 1:
        return n
    if n not in memo:
        memo[n] = fib_memo(n-1, memo) + fib_memo(n-2, memo)
    return memo[n]
# Time: O(n), Space: O(n)

3. Tabulation (Bottom-Up DP) - вЬЕ Most EfficientвАЛ

def fib_tab(n):
    if n <= 1:
        return n
    
    # 1. State: dp[i] is the i-th Fibonacci number
    dp = [0] * (n + 1)
    
    # 2. Base Cases
    dp[0] = 0
    dp[1] = 1
    
    # 4. Order: 2 to n
    for i in range(2, n + 1):
        # 3. Recurrence Relation
        dp[i] = dp[i-1] + dp[i-2]
    
    # 5. Final Answer
    return dp[n]
# Time: O(n), Space: O(n)

Visualization:

i=2: dp[2] = 1 + 0 = 1
dp = [0, 1, 1, 0, 0, 0]
i=3: dp[3] = 1 + 1 = 2
dp = [0, 1, 1, 2, 0, 0]
i=4: dp[4] = 2 + 1 = 3
dp = [0, 1, 1, 2, 3, 0]
i=5: dp[5] = 3 + 2 = 5

4. Space Optimized - вЬЕ O(1) SpaceвАЛ

def fib_optimized(n):
    if n <= 1:
        return n
    prev2, prev1 = 0, 1  # F(0), F(1)
    for i in range(2, n + 1):
        current = prev1 + prev2
        prev2, prev1 = prev1, current
    return prev1
# Time: O(n), Space: O(1)

рЯПЧпЄП DP Framework: 5-Step ProcessвАЛ

Step 1: Define the StateвАЛ

What does dp[i] represent?

Example: dp[i] = i-th Fibonacci number

Step 2: Define the Base CasesвАЛ

What are the smallest problems with known solutions?

Example: dp[0] = 0, dp[1] = 1

Step 3: Define the Recurrence Relation (What are the choices?)вАЛ

How does dp[i] depend on smaller states?

Example: dp[i] = dp[i-1] + dp[i-2]

Step 4: Determine Order of Computation (What is the transition?)вАЛ

Should we go bottom-up or top-down?

Example: Compute from dp[0] to dp[n]

Step 5: Return the AnswerвАЛ

Which state contains the final answer?

Example: Return dp[n]

рЯІ© Example 2: Climbing StairsвАЛ

Problem: You can climb 1 or 2 steps at a time. How many ways to reach the top?вАЛ

Observation: This is Fibonacci! ways(n) = ways(n-1) + ways(n-2)

n = 1: [1] вЖТ 1 way
n = 2: [1,1], [2] вЖТ 2 ways
n = 3: [1,1,1], [1,2], [2,1] вЖТ 3 ways
n = 4: вЖТ 5 ways

DP Solution:вАЛ

def climbStairs(n):
    if n <= 2:
        return n
    dp = [0] * (n + 1)
    dp[1] = 1  # 1 way to reach step 1
    dp[2] = 2  # 2 ways to reach step 2
    for i in range(3, n + 1):
        dp[i] = dp[i-1] + dp[i-2]
    return dp[n]

# Space optimized:
def climbStairs_opt(n):
    if n <= 2:
        return n
    prev2, prev1 = 1, 2
    for _ in range(3, n + 1):
        current = prev1 + prev2
        prev2, prev1 = prev1, current
    return prev1

State & Transition:вАЛ

  • State: dp[i] = number of ways to reach step i
  • Transition: dp[i] = dp[i-1] + dp[i-2]
graph LR
    S0 --> S1
    S0 --> S2
    S1 --> S2
    S1 --> S3

Visual Explanation:

To reach step i:
1. Come from step i-1 (take 1 step)
2. Come from step i-2 (take 2 steps)
Total ways(i) = ways(i-1) + ways(i-2)

рЯІ© Example 3: Unique Paths (2D DP)вАЛ

Problem: Robot at top-left, can only move right or down. How many unique paths to bottom-right?вАЛ

Grid: 3√Ч3
S вЖТ вЖТ 
вЖУ   вЖШ
вЖУ вЖШ E
Answer: 6 paths

DP Solution:вАЛ

def uniquePaths(m, n):
    # dp[i][j] = ways to reach cell (i,j)
    dp = [[0] * n for _ in range(m)]
    
    # Base cases
    for i in range(m):
        dp[i][0] = 1  # First column
    for j in range(n):
        dp[0][j] = 1  # First row
    
    # Fill DP table
    for i in range(1, m):
        for j in range(1, n):
            dp[i][j] = dp[i-1][j] + dp[i][j-1]
    
    return dp[m-1][n-1]

  • dp[i][j] = dp[i-1][j] + dp[i][j-1]

рЯТ∞ Example 4: Coin Change (Classic DP)вАЛ

Problem: Minimum coins to make amountвАЛ

Coins: [1, 2, 5]
Amount: 11
Answer: 3 (5 + 5 + 1)

State:вАЛ

dp[x] = minimum coins needed to make amount x

Transition:вАЛ

dp[x] = min(dp[x - coin] + 1)

flowchart TD
    A[Amount x]
    A -->|Try coin 1| B[x-1]
    A -->|Try coin 2| C[x-2]
    A -->|Try coin 5| D[x-5]

DP Solution:вАЛ

def coinChange(coins, amount):
    # dp[i] = min coins to make amount i
    dp = [float('inf')] * (amount + 1)
    dp[0] = 0  # 0 coins needed for amount 0
    
    for i in range(1, amount + 1):
        for coin in coins:
            if coin <= i:
                dp[i] = min(dp[i], dp[i - coin] + 1)
    
    return dp[amount] if dp[amount] != float('inf') else -1

Step-by-step (coins=[1,2,5], amount=11):

dp[0] = 0
dp[1] = min(dp[0]+1) = 1
dp[2] = min(dp[1]+1, dp[0]+1) = min(2,1) = 1
dp[3] = min(dp[2]+1, dp[1]+1) = min(2,2) = 2
...
dp[11] = min(dp[10]+1, dp[9]+1, dp[6]+1) = min(2+1, 3+1, 2+1) = 3

рЯФН How to Identify DP ProblemsвАЛ

Ask These Questions:вАЛ

  1. Can it be broken into subproblems?
  2. Do subproblems overlap? (Would recursion repeat calculations?)
  3. Is there optimal substructure? (Can we build optimal solution from optimal subproblems?)

Common DP Problem Patterns:вАЛ

1. "Count the number of ways..." вЖТ Often DP
2. "Find minimum/maximum..." вЖТ Often DP
3. "Find if possible..." вЖТ Often DP
4. Grid/matrix problems вЖТ Often 2D DP
5. String problems вЖТ Often 2D DP

How to Recognize a DP Problem (Interview Gold в≠Р)вАЛ

Look for these keywords:

  • "maximum / minimum"
  • "number of ways"
  • "longest / shortest"
  • "can you reach"
  • "optimal"

рЯЫ†пЄП DP Implementation PatternsвАЛ

Pattern 1: Fibonacci StyleвАЛ

# dp[i] depends on dp[i-1], dp[i-2]
dp = [0] * (n+1)
dp[0], dp[1] = base_cases
for i in range(2, n+1):
    dp[i] = dp[i-1] + dp[i-2]

Pattern 2: Grid DPвАЛ

# dp[i][j] depends on dp[i-1][j], dp[i][j-1]
dp = [[0]*cols for _ in range(rows)]
# Initialize first row/col
for i in range(rows):
    for j in range(cols):
        dp[i][j] = dp[i-1][j] + dp[i][j-1]

Pattern 3: Knapsack StyleвАЛ

# dp[i][w] = max value with first i items and weight w
dp = [[0]*(W+1) for _ in range(n+1)]
for i in range(1, n+1):
    for w in range(1, W+1):
        if weight[i-1] <= w:
            dp[i][w] = max(dp[i-1][w], 
                          dp[i-1][w-weight[i-1]] + value[i-1])
        else:
            dp[i][w] = dp[i-1][w]

Pattern 4: LCS (Longest Common Subsequence)вАЛ

# dp[i][j] = LCS of first i chars of A and j chars of B
dp = [[0]*(m+1) for _ in range(n+1)]
for i in range(1, n+1):
    for j in range(1, m+1):
        if A[i-1] == B[j-1]:
            dp[i][j] = dp[i-1][j-1] + 1
        else:
            dp[i][j] = max(dp[i-1][j], dp[i][j-1])

рЯЪ® Common Mistakes & How to Avoid ThemвАЛ

Mistake 1: Not defining state clearlyвАЛ

Fix: Write down what dp[i] means before coding

Mistake 2: Wrong base casesвАЛ

Fix: Test with smallest inputs (n=0, n=1)

Mistake 3: Off-by-one errorsвАЛ

Fix: Use 0-indexed or 1-indexed consistently

Mistake 4: Forgetting to initializeвАЛ

Fix: Always initialize DP array with appropriate values

Mistake 5: Wrong recurrence relationвАЛ

Fix: Draw small examples to verify logic

рЯФД Memoization vs TabulationвАЛ

Memoization (Top-Down)вАЛ

def solve(n, memo={}):
    if n in memo: return memo[n]
    if base_case: return value
    memo[n] = solve(n-1) + solve(n-2)
    return memo[n]

Pros:

  • More intuitive (like recursion)
  • Only computes needed states

Cons:

  • Recursion depth limit
  • Function call overhead

Tabulation (Bottom-Up)вАЛ

dp = [0]*(n+1)
for i in range(1, n+1):
    dp[i] = dp[i-1] + dp[i-2]

Pros:

  • No recursion limit
  • Often faster
  • Easier to optimize space

Cons:

  • Might compute unnecessary states
  • Less intuitive initially

рЯОУ DP Problem-Solving ChecklistвАЛ

Before Coding:вАЛ

  • Identify this is a DP problem
  • Define state variables clearly
  • Write recurrence relation on paper
  • Determine base cases
  • Decide memoization vs tabulation

While Coding:вАЛ

  • Initialize DP table properly
  • Handle edge cases
  • Verify recurrence with small example
  • Check indices/off-by-one errors

After Coding:вАЛ

  • Test with given examples
  • Test with edge cases (empty, minimum size)
  • Consider space optimization
  • Analyze time/space complexity

# After: Keep only last 2 values
prev2, prev1 = 0, 1
for i in range(2, n+1):
    current = prev1 + prev2
    prev2, prev1 = prev1, current

2. Reuse Input Array (if allowed)вАЛ

# Modify input array in place as DP table

3. Alternate Rows in 2D DPвАЛ

# When dp[i][j] only depends on previous row
dp_prev = [0]*n
dp_curr = [0]*n
for i in range(1, m):
    for j in range(1, n):
        dp_curr[j] = dp_prev[j] + dp_curr[j-1]
    dp_prev, dp_curr = dp_curr, dp_prev

def dp_solution(inputs):

рЯІ† Mental Models for DPвАЛ

Model 1: The "Building Up" ModelвАЛ

Think: "If I know solutions for all smaller problems, how do I solve the current one?"

Model 2: The "Decision Tree" ModelвАЛ

At each step, you make a choice. DP finds the optimal path through the tree.

Model 3: The "Caching" ModelвАЛ

Remember expensive computations so you don't repeat them.

Model 4: The "State Machine" ModelвАЛ

Your state represents where you are in the problem. Transitions represent decisions.
