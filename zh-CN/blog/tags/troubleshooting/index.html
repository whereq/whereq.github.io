Best Practices of Linux System Performance Analysis and Tuning with CLI

Introduction​

Monitoring and tuning Linux system performance is crucial for maintaining the health and efficiency of your infrastructure. Command line tools like top, htop, iotop, sar, ps, free, vmstat, and netstat provide powerful capabilities for analyzing and optimizing system performance. This article provides a comprehensive guide to the best practices of using these tools, covering various scenarios such as checking CPU and memory usage, monitoring processes, and troubleshooting real production issues.

Basic System Monitoring​

Check CPU Usage​

To check CPU usage, use the top command:

top

This command displays real-time CPU and memory usage, along with a list of running processes.

Check Memory Usage​

To check memory usage, use the free command:

free -h

This command displays the total, used, and free memory in a human-readable format.

Check Disk I/O​

To check disk I/O, use the iostat command:

iostat -x 1

This command displays detailed I/O statistics, including read and write rates.

Check Network Traffic​

To check network traffic, use the iftop command:

iftop -i eth0

This command displays real-time network traffic on the eth0 interface.

Process Monitoring​

List Running Processes​

To list all running processes, use the ps command:

ps aux

This command displays a detailed list of all running processes.

Check Process Status​

To check the status of a specific process, use the ps command with the process ID (PID):

ps -p <PID> -o pid,ppid,cmd,%cpu,%mem,state

This command displays the status of the process with the specified PID.

Kill a Process​

To kill a process, use the kill command with the process ID (PID):

kill <PID>

This command sends a termination signal to the process with the specified PID.

Advanced Performance Tuning​

Top​

To monitor system performance in real-time, use the top command:

top

This command provides a dynamic view of system resources and running processes.

Htop​

To monitor system performance with a more user-friendly interface, use htop:

htop

This command provides a colorful and interactive view of system resources and running processes.

Iotop​

To monitor disk I/O in real-time, use the iotop command:

iotop

This command displays real-time disk I/O statistics for running processes.

Sar​

To collect and report system activity, use the sar command:

sar -u 1 5

This command collects CPU usage statistics every second for 5 seconds.

Troubleshooting Real Production Issues​

High CPU Usage​

To identify processes causing high CPU usage, use top or htop:

top

Sort processes by CPU usage by pressing P.

High Memory Usage​

To identify processes causing high memory usage, use top or htop:

top

Sort processes by memory usage by pressing M.

High Disk I/O​

To identify processes causing high disk I/O, use iotop:

iotop

This command displays real-time disk I/O statistics for running processes.

Network Latency​

To diagnose network latency, use ping and traceroute:

ping google.com
traceroute google.com

These commands help identify network latency and routing issues.

Conclusion​

Monitoring and tuning Linux system performance is essential for maintaining the health and efficiency of your infrastructure. By mastering the various options and scenarios covered in this article, you can effectively use command line tools like top, htop, iotop, sar, ps, free, vmstat, and netstat to analyze and optimize system performance.

References​

  2. htop Man Page
  3. iotop Man Page
  4. sar Man Page
  5. ps Man Page
  6. free Man Page
  7. iostat Man Page
  8. iftop Man Page
  9. ping Man Page
  10. traceroute Man Page

By following these best practices, you can leverage these system performance tools to their full potential and efficiently manage your monitoring and tuning tasks.

Introduction​

In a K8S cluster, PodA is experiencing issues when using a service name to access PodB. PodA is on node1, and PodB is on node2. This article will walk through the troubleshooting steps to identify and resolve the issue.

Problem Description​

PodA is unable to access PodB using the service name, and the request is failing. PodA is on node1, and PodB is on node2.

Troubleshooting Steps​

Step 1: Capture Network Traffic​

First, use tcpdump to capture network traffic and observe if there are any anomalies:

[root@node1 ~]# tcpdump -n -i ens192 port 50300  
...  
13:48:17.630335 IP 177.177.176.150.distinct -> 10.96.22.136.50300: UDP, length 214  
13:48:17.630407 IP 192.168.7.21.distinct -> 10.96.22.136.50300: UDP, length 214  
...

From the captured packets, the source address and port are 177.177.176.150:50901, and the destination address and port are 10.96.22.136:50300. The destination IP 10.96.22.136 is the IP address obtained by PodA when using the server-svc service name.

Step 2: Verify Service IP​

Check if the IP address 10.96.22.136 is correct:

[root@node1 ~]# kubectl get pod -A -owide|grep server  
ss  server-xxx-xxx  1/1  Running 0 20h  177.177.176.150  node1  
ss  server-xxx-xxx  1/1  Running 0 20h  177.177.254.245  node2  
ss  server-xxx-xxx  1/1  Running 0 20h  177.177.18.152   node3  
[root@node1 ~]# kubectl get svc -A -owide|grep server  
ss  server-svc  ClusterIP  10.96.182.195 <none>  50300/UDP

The source IP is correct, but the destination IP does not match the expected server-svc IP 10.96.182.195.

Step 3: Check DNS Resolution​

Since K8S uses CoreDNS for service discovery starting from v1.13, verify if the DNS resolution is correct inside PodA:

[root@node1 ~]# kubectl exec -it -n ss server-xxx-xxx -- cat /etc/resolve.conf  
nameserver 10.96.0.10  
search ss.svc.cluster.local svc.cluster.local cluster.local  
options ndots:5  
  
[root@node1 ~]# kubectl exec -it -n ss server-xxx-xxx -- nslookup server-svc  
Server: 10.96.0.10  
  
Name: ss  
Address: 10.96.182.195

The DNS resolution is correct, and PodA can resolve server-svc to 10.96.182.195.

Step 4: Investigate Service IP​

Check if 10.96.22.136 is associated with any other service, iptables rules, or routes:

[root@node1 ~]# kubectl get svc -A -owide|grep 10.96.22.136  
  
[root@node1 ~]# iptables-save|grep 10.96.22.136  
  
[root@node1 ~]# ip route|grep 10.96.22.136

The IP 10.96.22.136 does not exist in the cluster.

Step 5: Verify Outgoing Traffic from Pod​

Check the destination IP when the traffic leaves PodA:

[root@node1 ~]# ip route|grep 177.177.176.150  
177.177.176.150 dev cali9afa4438787 scope link  
  
[root@node1 ~]# tcpdump -n -i cali9afa4438787 port 50300  
...  
14:16:40.821511 IP 177.177.176.150.50902 ->  10.96.22.136.50300:  UDP, length 214  
...

The destination IP is incorrect when leaving PodA.

Step 6: Simulate Traffic from Pod​

Simulate a UDP packet from PodA using nc to verify if the destination IP is correct:

[root@node1 ~]# kubectl exec -it -n ss server-xxx-xxx -- echo “test” | nc -u server-svc 50300 -p 9999  
  
[root@node1 ~]# tcpdump -n -i cali9afa4438787 port 50300  
...  
15:46:45.871580 IP 177.177.176.150.50902 ->  10.96.182.195.50300:  UDP, length 54  
...

The simulated traffic has the correct destination IP.

Step 7: Restart Pod​

Restart PodA to see if the issue is resolved:

[root@node1 ~]# docker ps |grep server-xxx-xxx | grep -v POD |awk '{print $1}' |xargs docker restart  
  
[root@node1 ~]# tcpdump -n -i ens192 port 50300  
...  
15:58:17.150535 IP 177.177.176.150.distinct -> 10.96.182.195.50300:  UDP, length 214  
15:58:17.150607 IP 192.168.7.21.distinct  ->  10.96.182.195.50300:   UDP, length 214  
...

The issue is resolved after restarting PodA.

Step 8: Identify Java DNS Caching​

Investigate if Java DNS caching is causing the issue:

In Java, DNS caching is used to improve performance by storing DNS lookup results. When InetAddress is used to resolve a domain name, the result is cached by the JVM. This cache can cause stale DNS entries to be used.

To resolve this, adjust the DNS caching settings:

  1. Set the networkaddress.cache.ttl property in the Java security settings.
  2. Modify the java.security file to set the networkaddress.cache.negative.ttl property.

Solution​

Adjust the DNS caching settings in the business application based on the business scenario.

Conclusion​

This article detailed the troubleshooting steps to identify and resolve an issue where PodA was unable to access PodB using a service name in a K8S cluster. The root cause was identified as Java DNS caching, and the issue was resolved by adjusting the DNS caching settings.

func (ds *dockerService) getIP(podSandboxID string, sandbox *dockertypes.ContainerJSON) string {
    if sandbox.NetworkSettings == nil {
        return ""
    }
    if networkNamespaceMode(sandbox) == runtimeapi.NamespaceMode_NODE {
        // For sandboxes using host network, the shim is not responsible for
        // reporting the IP.
        return ""
    }

    // Don't bother getting IP if the pod is known and networking isn't ready
    ready, ok := ds.getNetworkReady(podSandboxID)
    if ok && !ready {
        return ""
    }

    ip, err := ds.getIPFromPlugin(sandbox)
    if err == nil {
        return ip
    }
    
    if sandbox.NetworkSettings.IPAddress != "" {
        return sandbox.NetworkSettings.IPAddress
    }
    if sandbox.NetworkSettings.GlobalIPv6Address != "" {
        return sandbox.NetworkSettings.GlobalIPv6Address
    }

    // Error log here
    klog.Warningf("failed to read pod IP from plugin/docker: %v", err)
    return ""
}

Pod Deletion Process​

The deletion of a Pod in Kubernetes follows these steps:

  1. API Server DELETE Call: The kubectl delete command triggers a DELETE request to the Kubernetes API server with a default grace period of 30 seconds.
  2. Update Pod Metadata: The Pod's metadata is updated with DeletionTimestamp and DeletionGracePeriodSeconds fields, but the Pod is not yet deleted from etcd.
  3. Kubelet Event Handling: The kubelet component listens for Pod updates and initiates the killPod() method.
  4. Second DELETE Call: The kubelet makes a second DELETE request to the API server with a grace period of 0 seconds.
  5. Pod Deletion from etcd: The API server deletes the Pod from etcd.

Introduction​

In Kubernetes environments, monitoring resource usage is crucial for maintaining optimal performance and resource allocation. This article delves into a specific case where a user observed high memory usage by a Harbor pod on one of the nodes, leading to a discrepancy between kubectl top and docker stats results. We will analyze the cause, verify the memory calculation methods, and provide a solution to accurately monitor memory usage.

Problem Background​

A user observed that one of the nodes in their Kubernetes cluster showed high memory usage by a Harbor pod, approximately 3.7G, which seemed unusually high compared to other pods.

[root@node02 ~]# kubectl get node -owide
NAME    STATUS   ROLES    AGE   VERSION    INTERNAL-IP   EXTERNAL-IP

[root@node02 ~]# kubectl top pod -A | grep harbor

kube-system         harbor-master2-ncvb8                   8m            3781Mi
kube-system         harbor-master3-2gdsn                   14m           227Mi

Cause Analysis​

To understand the memory usage, the user compared the results from kubectl top and docker stats. Theoretically, docker stats should provide a more accurate result.

Using docker stats​

The total memory usage from docker stats was around 140M, significantly lower than the 3.7G reported by kubectl top.

Verification of Memory Calculation​

The discrepancy arises from the different methods used by kubectl top and docker stats to calculate memory usage.

  • kubectl top Calculation: memory.usage_in_bytes - inactive_file
  • docker stats Calculation: memory.usage_in_bytes - cache

If the cache is large, kubectl top will report a higher memory usage.

Verifying with memory_stats​

To verify, the user fetched the memory_stats data using Docker's API:

Sample memory_stats data:

The high cache usage in Harbor was primarily due to the registry component, which handles Docker image storage and involves extensive read/write operations on image layers.

Solution​

To address the discrepancy, the user was informed that kubectl top includes cache usage, which can be misleading. The cache, being non-critical, can be reclaimed by the system during memory pressure or manually released. The user was advised to use docker stats for a more accurate representation of actual memory usage.

Conclusion​

This case study highlights the importance of understanding the different methods used by Kubernetes and Docker to calculate memory usage. By verifying the calculations and understanding the role of cache, users can make more informed decisions about resource management in their Kubernetes clusters. Using docker stats provides a more accurate view of memory usage, especially in scenarios where cache usage is significant.

TCP State Transition Diagram​

graph TD
    CLOSED -->|Client sends SYN| SYN_SENT
    SYN_SENT -->|Server responds with SYN+ACK| SYN_RECV
    SYN_RECV -->|Client sends ACK| ESTABLISHED
    ESTABLISHED -->|Connection closed by either party| FIN_WAIT_1
    FIN_WAIT_1 -->|ACK received| FIN_WAIT_2
    FIN_WAIT_2 -->|FIN received| TIME_WAIT
    TIME_WAIT -->|Timeout| CLOSED
    FIN_WAIT_1 -->|FIN received| CLOSING
    CLOSING -->|ACK received| TIME_WAIT
    SYN_RECV -->|Server sends FIN| LAST_ACK
    LAST_ACK -->|Client sends ACK| CLOSED
